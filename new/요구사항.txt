

정식 원본 베타플라이트 컨피규레이터를 다운받아 
사용자 요구에 맞게 수정하여 사용하려고 한다. 여러 문제가 존재하니
해결 방법을 찾아야 한다.

표고 프로필 문제

코드 위치 https://github.com/saydals/bf/tree/master/src/components/tabs/FlightPlan

정식 원본은 해발고도 기준으로 기체의 고도나 속도를 표시한다.
만약 해발고도 100미터에서 기체를 200미터 높이로 띄우겠다고 
입력하면 원본의 결과는 둘다 해발고도를 표시하므로 100미터만 상승한다.
문제는 해발고도 1000미터에서 200미터 높이 띄우겠다고 했을때
정식 원본에서의 기체의 높이는 1200미터로 입력되어 원치 않는 경과가 나타난다.


수정 버전은 모두 WP1을 기준으로 상대 고도 기준이다.
상대고도를 사용하는 이유는 대부분 사용하는 RC가 기압계 기반이며
출발지의 고도를 0으로 표시하고 있기 때문이다.

하지만 지도 정보는 해발고도를 제공하므로 변환하여 사용해야한다.

WP1은 홈포인트와 같은 해발고도라고 가정한다 ( 사용자 주의 사항 )

문제 1

WP포인트를 드래그시 바로 따라오지 않는다
마우스 위치로 포인트가 즉시 따라오는것이 맞다.

좌우 드래그시 개선
상하와 좌우를 동시에 출력 하지 않는다는게 전제조건.
좌우로 움직일때는 포인트가 살짝 좌우로 움직여 ( 많이 끌어도 조금만 움직임 )
속도 조절중이라는 것을 포인트가 움직이는걸로 표시할 수 있다.
처음 터치시 좌우인지 상하인지 판별하여 한 동작만 한다.


문제 2

상대고도이므로 WP1이 고지일경우 표시되는 지표고도 심지어 최대 지면 표고가 0보다 작을 수 있다는 것을 허용해야한다.
또 WP2 부터는 기체의 상대고도가 0보다 작을수 있다.
음수 고도를 처리하는지 확인해야 한다. 
기압계 기반 고도계는 음수를 표시하는것은 확실하며 OSD 고도값도 음수로 표시된다.
다만 음수로 설정한 고도가 기체 펌웨어에서 처리해 주는지는 확실치 않다. 
https://github.com/saydals/autopilot/tree/autopilot/src/main/flight gps_rescue.c 에서 확인해봐야 한다.
만약 음수 고도를 처리해 주지 않는다면 WP2의 고도는 WP1의 지표고도 ( 상대지표고도 0 )에 제한된다. ( 현재 그렇게 처리하고 있음 )
음수 고도를 처리해 준다면 WP2 부터는 음수 고도를 가질 수 있어야 한다.

문제 3

지면표고 표시가 현재 평균 해발 고도를 표시하는것 같다.
원래는 포인트를 선택했을때의 지면표고가 표시되었다.
이미 평균 지면 표고는 그래프 상에서 표시 하고 있으므로
각 WP 포인트 위치에서 상대지면표고를 표시하는것이 맞다.
이것으로 WP포인트의 위치가 사용자가 날리는 곳보다 높은지 낮은지 알수 있다.
제일 왼쪽에 표시되는 고도는 해발고도로 여겨지는데 수정하지 않는것이 좋다.
다만 해발고도라고 표시해주는것이 혼란 방지.
지면표고 -> 상대지면 표고
최대 지면표고 -> 상대 최대 지면표고로 표시

문제 4

WP포인트를 클릭하면 툴팁이 표시되는데 현재 " 고도 : 00 m AGl " -> " 상대고도 : 00 m " 라고 표시한다.
또 포인트를 움직이는 중에 고도 변화에 맞춰 실시간으로 속도와 고도를 표시해야 한다.
속도의 변화는 포인트가 움직인 양이 아니고 중심에서 벗어나 좌우로 벗어나 있는 시간에 따라
변화한다. 변화량은 기존코드대로 사용하면 되지만 툴팁에 속도표시가 너무 늦게 갱신된다.
waypoint list에서 보여주는 속도와 표고프로필에서 보이는 속도가 갱신 속도 차이가 크다.


위 문제 지적중에 논리적으로 오류가 있으면 사용자에게 문제 제기한다. 추정하지 않는다.

모든 문제의 해결책을 찾고 코드 수정할 부분을 찾는다.
최종 문서는 코딩 AI에게 시킬 작업 지시서 형태의 MD 파일로 출력한다.

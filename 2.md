11. 웹게시(GitHub Pages) 배포 방법:
    - 저장소: https://github.com/saydals/bf
    - 웹 주소: https://saydals.github.io/bf/
    - 배포 방식: GitHub Actions 자동 배포
    - workflow 파일: .github/workflows/gh-pages.yml
    - 배포 과정:
      1. master 브랜치에서 변경사항을 커밋하고 push
      2. GitHub Actions가 자동 실행되어 npm ci → npm run build 실행
      3. 빌드 결과물(src/dist)이 gh-pages 브랜치에 자동 배포됨
      4. GitHub Pages가 gh-pages 브랜치를 서빙 (약 2~10분 소요)
    - 주의: gh-pages 브랜치에 직접 push하지 말 것 (GitHub Actions가 관리)
    - 빌드 필수 파일: src/js/msp/debug/ (git add -f 필요, .gitignore로 무시됨)

12. GitHub Pages 배포가 안 될 때 확인 사항:

    - GitHub 저장소 → Actions 탭에서 workflow 실행 내역 확인
    - workflow 실패 시: npm install, vite build를 로컬에서 먼저 테스트
    - src/js/msp/debug/ 디렉토리가 누락되면 빌드 실패 (git add -f로 강제 추가)
    - 배포 후 CDN 캐시 때문에 최대 10분까지 이전 버전이 보일 수 있음
    - 브라우저 강력 새로고침(Ctrl+Shift+R)으로 캐시 초기화

13. 이 저장소는 웹빌드만 한다. "빌드"라는 명령어는 오직 **웹빌드(vite build → dist/ 정적 웹 번들 생성)** 만을 의미한다. 안드로이드 APK 생성(android:apk 등)은 빌드에 포함되지 않으며, 제5항에 따라 절대 하지 않는다.

14. 빌드(또는 웹빌드) 명령시 해야 할 일:
    - 웹빌드 명령: `npm run build` (내부적으로 `vite build` 실행, dist/ 생성)
    - 빌드 전 필수 준비:
        1. `npm install` (또는 `pnpm install`) — 의존성 설치
        2. Android SDK 불필요 (웹빌드는 웹 번들만 생성하므로 네이티브 도구 불필요)
        3. 공식 저장소에서 누락된 보조 파일 복사가 필요할 수 있음:
            - `libraries/`
            - `src/js/msp/debug/` (git add -f 필요, .gitignore로 무시됨)
    - 빌드 결과 확인:
        - `dist/` 디렉토리 생성 여부 확인
        - `npm run preview` 로 로컬에서 빌드 결과 동작 점검 (http://localhost:8080)
    - 주의: 웹빌드는 APK 생성과 무관하므로 gradlew/cap sync 등 안드로이드 단계를 절대 수행하지 않는다.
